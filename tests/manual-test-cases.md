# 猪猪食堂 手动测试用例

## 测试环境
- 公网环境：`https://zhuzhu-canteen.vercel.app`
- 局域网环境：在 Mac 上执行 `npm run dev -- --hostname 0.0.0.0`，再用同一 Wi-Fi 下的手机访问终端显示的局域网地址
- 或电脑浏览器访问 `http://localhost:3000`

> 发布前须在两部不同手机上记录结果。未实际操作的项目标记为 `PENDING`，不得以桌面浏览器结果代替。

## 一、主流程测试

### TC-01：拍照记录菜品（完整流程）
1. 打开首页，确认在「菜品记录」tab
2. 点击「拍照记录一道菜」
3. 选择拍照或从相册选择照片
4. 确认照片预览显示正常
5. 在输入框中填写：菜品名称（如"可乐鸡翅"）
6. 选择分类（如"肉类"）
7. 填写食材清单
8. 填写做法步骤
9. 点击「保存到菜单库」
10. **预期**：保存成功，弹窗关闭，菜品出现在最近记录列表中

### TC-02：AI 智能识别（公网需有效密钥，局域网可用 Ollama）
1. 点击「拍照记录一道菜」→ 选择一张菜品图片
2. 点击「AI 智能识别（可选）」
3. 等待识别完成
4. **预期**：最多返回 3 个菜名候选、分类建议和照片中可见食材；不自动生成或覆盖菜谱用量与做法
5. 选择候选或手动输入菜名，确认后可修改并保存

### TC-03：AI 识别失败后的手动输入
1. 选择一张菜品图片
2. 点击「AI 智能识别（可选）」
3. 如果模型超时或密钥无效，**预期**：提示识别失败并允许手动填写，已上传的照片仍显示且不丢失
4. 手动填写菜品信息 → 点击保存
5. **预期**：保存成功，菜品出现在菜单库

### TC-04：浏览菜单库
1. 切换到「菜单库」tab
2. **预期**：显示所有已记录的菜品（按分类排列）
3. 点击分类按钮（如"🥩 肉类"）
4. **预期**：只显示该分类的菜品
5. 在搜索框输入菜品名称
6. **预期**：只显示匹配的菜品

### TC-05：查看菜品详情
1. 在菜单库中点击任意菜品
2. **预期**：弹出详情页，显示菜品图片、名称、分类、食材清单、做法步骤
3. 点击「早餐/午餐/晚餐」按钮
4. **预期**：菜品加入今日菜单，弹窗关闭

### TC-06：随机推荐菜品
1. 切换到「今天吃点啥」tab
2. **预期**：随机显示一道菜
3. 点击「换一道」→ **预期**：显示不同菜品
4. 点击分类标签（如"🥬 青菜"）→ **预期**：只从该分类随机
5. 点击「🌅 早餐」按钮 → **预期**：菜品加入今日早餐列表
6. 切换到下方「今日菜单」，**预期**：早餐栏显示刚添加的菜品

### TC-07：查看制作历史
1. 切换到「历史」tab
2. **预期**：显示常做菜品排行（柱状图）+ 最近点菜记录
3. 点击某条记录 → **预期**：跳转到对应的菜品详情

### TC-08：删除今日菜单中的菜品
1. 在「今天吃点啥」→「今日菜单」中
2. 点击菜品右侧的「✕」按钮
3. **预期**：菜品从该餐次中移除

## 二、边界条件测试

### TC-09：新用户首次使用（无数据）
1. 清空数据库后打开应用
2. **预期**：
   - 「菜品记录」tab 显示"还没有记录菜品，快去拍第一张吧 📸"
   - 「菜单库」tab 显示"还没有记录菜品，先去拍一张吧 📸"
   - 「今天吃点啥」tab 随机推荐区显示"该分类暂无菜品 先去拍一张吧 📸"
   - 「历史」tab 显示"还没有点菜记录，去排个菜单吧 📅"
3. 所有 tab 切换正常，无白屏，无报错

### TC-10：空字段校验
1. 点击「拍照记录一道菜」→ 选择照片
2. 不填菜品名称，直接点保存
3. **预期**：保存按钮不显示（因为 `canSave = editName.trim() && file`）
4. 去掉照片，只填菜品名称
5. **预期**：保存按钮不显示

### TC-11：超长文本输入
1. 菜品名称输入 50 个汉字
2. 食材清单输入 50 行
3. 做法步骤输入 50 行
4. 点击保存
5. **预期**：保存成功，数据不丢失不截断

### TC-12：特殊字符输入
1. 菜品名称输入 `红烧排骨 <script>alert(1)</script>`
2. 点击保存
3. **预期**：保存成功（React 默认转义，无 XSS 风险）

### TC-13：重复保存同一菜品
1. 输入已在饭盆中存在的菜名并尝试保存
2. **预期**：显示「已有」提示并在保存前拦截，不会创建重复记录

### TC-14：无分类菜品
1. 新增菜品时，categoryId 填 null 或 0
2. **预期**：保存成功，菜单库中该菜品在"全部"分类下显示，但不属于任何具体分类过滤

### TC-15：空食材/空步骤
1. 新增菜品时，食材和步骤都留空
2. **预期**：保存成功，详情页不显示食材和步骤区域

### TC-16：快速切换 Tab
1. 快速连续切换 4 个 tab（菜品记录→菜单库→今天吃点啥→历史→菜品记录）
2. **预期**：每个 tab 正常渲染，无闪烁、无崩溃

### TC-17：添加已删除的菜品到菜计划
1. 数据库中有菜品 A
2. 将菜品 A 加入今日早餐
3. 在「今日菜单」中删除早餐中的菜品 A
4. 再在「随机推荐」中刷到菜品 A，加入午餐
5. **预期**：菜品 A 正常加入午餐列表

### TC-18：同一餐次添加多个菜品
1. 对午餐连续添加 3 道不同菜品
2. **预期**：3 道菜都显示在午餐栏中

### TC-19：跨天菜单隔离
1. 今天加入菜品到午餐
2. 修改请求参数 date 为昨天，查询昨天菜单
3. **预期**：昨天的菜单不包含今天添加的菜品

### TC-20：菜品已删除但菜计划仍引用
1. 数据库中有菜品 B，已加入今日午餐
2. 直接从数据库删除菜品 B（不通过前端）
3. 刷新「今天吃点啥」→「今日菜单」
4. **预期**：午餐栏中菜品 B 不显示（因为 `getDish` 返回 undefined 时渲染 null）

## 三、API 与数据库测试安全

```bash
# 单元和配置测试，不访问生产 API
npm test

# 仅使用临时本地数据库
npm run test:db
```

`npm run test:api` 在隔离测试数据库和独立测试服务未配置前被刻意阻止。不要对使用正常 `.env.local` 的开发服务执行写入型 API 自动测试。

### 需手动冒烟验证的 API：
- `GET /api/dishes` - 列表、搜索、分类过滤
- `POST /api/dishes` - 创建（带数据、空数据、超长数据）
- `GET /api/categories` - 分类初始化
- `POST /api/plans` - 创建菜计划
- `GET /api/plans` - 查询菜计划
- `DELETE /api/plans` - 删除菜计划
- `GET /api/recipes/search` - 公共菜谱搜索
- `GET /api/recipes/:id` - 公共菜谱详情
- `GET/POST/DELETE /api/wishlist` - 心愿列表、新增与移除
- `GET /api/wishlist/completed` - 已完成心愿
- `GET /api/recommendations` - 饭盆与心愿的双源推荐
- `POST /api/recognize` - 最多 3 个识图候选

## 四、猪猪心愿单手动测试

### TC-21：心愿单入口与返回
1. 切换到「菜单」tab，确认标题右侧显示 44×44px 以上的爱心按钮。
2. 待完成心愿数为 0 时，确认爱心上没有数字 badge；数量大于 0 时才显示。
3. 点击爱心进入「猪猪心愿单」，确认底部四个 Tab 未新增入口且在全屏视图中隐藏。
4. 点击左上返回按钮，确认回到菜单页；在带刘海和底部手势条的设备上，内容不被安全区遮挡。

### TC-22：显式搜索与空结果
1. 在搜索框输入菜名但不提交，确认页面不会自动发起搜索、不会生成 AI 菜谱。
2. 点击「搜索」或按键盘提交，确认出现加载状态，然后显示真实菜谱结果。
3. 搜索不存在的菜名，确认显示「暂无匹配菜谱」。
4. 模拟搜索接口失败，确认显示错误信息且可以再次提交。

### TC-23：菜谱详情与加入心愿
1. 从搜索结果进入菜谱详情，确认显示采购清单、做法步骤和内容来源。
2. 点击「加入猪猪心愿单」，确认成功提示、按钮变为已加入且不可重复点击，菜单页 badge 数量更新。
3. 打开已经在心愿单的菜谱，确认加入按钮初始即禁用。
4. 打开已经做过的菜谱，确认显示「已经做过」标签。
5. 点击返回，确认回到此前的心愿单或已完成列表。

### TC-24：移除确认与已完成列表
1. 在待完成心愿卡片确认可见分类、加入日期和详情入口。
2. 点击「移除心愿」，确认先弹出二次确认；点击「再想想」不应移除。
3. 再次操作并点击「确认移除」，确认心愿从列表消失，返回菜单后 badge 更新。
4. 点击「已完成」，确认列表显示菜名与完成日期；有菜谱来源的记录可进入详情并返回。

### TC-25：心愿菜加入今日菜单不会完成心愿
1. 将一道未完成心愿菜加入早餐、午餐或晚餐。
2. 返回心愿单。
3. **预期**：该菜仍在「待完成」列表，数量不变，且没有新增完成记录。

### TC-26：保存成品并完成心愿的原子性
1. 为一道待完成心愿上传实际成品照，菜名与心愿相同。
2. 点击保存，在确认框中选择「移除并记为已完成」；快速连点确认按钮。
3. **预期**：只新增 1 条饭盆菜品和 1 条心愿完成记录，待完成数减 1，显示「心愿已完成 +1」动效。
4. 删除该饭盆菜品，**预期**：已完成心愿的名称、时间和图片快照仍保留。

### TC-27：随机推荐双数据源
1. 打开随机推荐，分别刷到饭盆菜和心愿菜。
2. **预期**：卡片明确显示「饭盆 · 做过 N 次」或「心愿单 · 还没做过」。
3. 如果同一菜同时存在于两个库，**预期**：推荐池中只出现一次。

### TC-28：两部手机图片与页面安全区
1. 在两部不同手机上分别打开公网和局域网页面，浏览首页、菜单库、推荐、日记、心愿单和菜谱详情。
2. **预期**：所有缩略图均能加载，页面可滚动到最底部，没有内容被底部 Tab 或手势条遮挡。
3. 分别记录机型、浏览器、网络、公网/局域网结果。

## 五、已知限制

1. **AI 识别**：公网环境需要有效的 Gemini 密钥；局域网可使用本机 Ollama/Qwen POC。模型结果只作为最多 3 个候选，不会自动确认菜名
2. **图片上传**：依赖 Vercel Blob，需要有效的 `BLOB_READ_WRITE_TOKEN`
3. **局域网访问**：手机和电脑需在同一 Wi-Fi，IP 可能随路由器分配而变化，以当次终端显示为准
4. **数据持久化**：数据库在 Turso Cloud，需要稳定的网络连接
5. **删除菜品**：删除为永久操作，有二次确认，但不提供回收站或撤销
