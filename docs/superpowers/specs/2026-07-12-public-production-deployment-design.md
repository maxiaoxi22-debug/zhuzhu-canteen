# 猪猪食堂公网生产部署设计

## 目标

把当前仅依赖家庭局域网访问的猪猪食堂升级为可通过固定 HTTPS 地址访问的公网生产版本，同时保留现有 Turso 数据库和 Vercel Blob 图片，不迁移或清空用户数据。

## 发布方案

继续使用项目已关联的 Vercel 项目 `zhuzhu-canteen`。这是当前改动最少、与 Next.js 16 兼容性最好、且已有账号和环境变量基础的方案。生产发布完成后使用项目固定的 `*.vercel.app` 地址，不使用带随机字符串的单次部署地址。

没有自有域名时，无法保证 `vercel.app` 在中国大陆所有运营商网络长期稳定；本次交付目标是消除对家庭局域网的依赖并提供可验证的公网 HTTPS 地址，不承诺大陆网络可用性 SLA。

## 代码改进范围

1. 修复新增菜品分类 ID 的索引偏移，确保肉类到其他六个分类分别保存为数据库 ID 1 到 6。
2. 保存菜品时检查 HTTP 响应；失败时保留表单并显示错误，不再误报成功。
3. 使用 Asia/Shanghai 本地日期生成今日菜单日期，避免凌晨被归入前一天。
4. 清理生产代码中的 ESLint 错误与无用引用，使发布前静态检查通过。
5. 不在本次范围内增加登录、迁移存储、购买域名或重做页面设计。

## 数据与请求流程

浏览器通过 Vercel HTTPS 地址访问 Next.js 页面和 `/api/*` 接口。接口继续使用现有环境变量连接 Turso；图片继续由服务端上传至 Vercel Blob；Gemini 识别失败时保留现有手动录入降级流程。

所有环境变量只配置在本地 `.env.local` 与 Vercel 项目环境中，不写入 Git。发布前只检查变量名称和线上接口结果，不输出密钥内容。

## 错误处理

- 数据库或网络保存失败：表单不关闭，展示可重试错误。
- 图片或 AI 识别失败：允许用户继续手动填写；已有图片 URL 时继续使用。
- 部署失败：不提升失败部署，保留当前可用生产版本，并根据构建日志修复后重新发布。

## 验证与发布

实施采用回归测试先行：先为分类映射、上海日期和保存失败行为补测试并确认测试能捕获现有问题，再修改代码。

发布前执行完整测试、ESLint 和生产构建。随后使用 Vercel CLI 发布 Production，并从公网地址验证：首页、菜品列表、分类、今日菜单查询均返回成功。最终交付固定公网地址与仍存在的外部服务限制。
