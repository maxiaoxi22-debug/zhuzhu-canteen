# 猪猪食堂 AI 识图、公共菜谱与心愿单设计规格

## 1. 目标与范围

本轮在不破坏现有饭盆、今日菜单、图片、编辑、删除和成长日记能力的前提下，新增三个相互隔离的产品域：

1. 公共菜谱库：导入固定版本的 HowToCook 菜谱文字内容，提供搜索和详情。
2. 猪猪心愿单：保存想做但尚未做过的菜，并永久保留完成快照。
3. AI 菜名识别：从成品照片返回最多三个候选菜名，不生成或伪造菜谱用量和步骤。

正式公网环境继续使用 Gemini；家庭局域网环境增加 Ollama `qwen3-vl:4b-instruct-q4_K_M` POC。真实账号系统保持为独立后续工作流，本轮仅为数据所有权预留字段，不混入登录开发。

## 2. 产品边界

### 2.1 公共菜谱库

公共菜谱包含菜名、别名、分类、人数、预计时间、食材及用量、制作步骤、来源和许可证。公共菜谱不代表用户做过，不能直接写入 `dishes`，也不能增加做过次数或成长统计。

首批数据来自 [Anduin2017/HowToCook](https://github.com/Anduin2017/HowToCook)，固定到导入时记录的 Git commit。导入脚本保留原始文件路径、来源 URL 和 Unlicense 许可证信息，并输出导入成功、跳过和解析失败统计。

### 2.2 猪猪心愿单

心愿单保存想做但尚未实际完成的菜。心愿通常关联公共菜谱；本轮不开放自建心愿菜，但数据结构允许 `recipe_id` 为空以便后续扩展。

加入今日菜单、查看菜谱、随机推荐命中或仅计划制作都不能完成心愿。只有用户上传实际成品照片并保存进饭盆，且主动确认后，心愿才会完成。

### 2.3 猪猪饭盆

现有 `dishes` 继续只保存真正做过的菜，包括用户成品照片、做过次数和历史关联。已有饭盆数据全部保留，不自动改名、不覆盖图片、不删除。明确匹配公共菜谱时，只补充可为空的 `recipe_id` 关联。

## 3. 信息架构与页面

底部导航保持现有四项，不增加新 Tab。

### 3.1 菜单页

菜单页标题区域右上角增加爱心入口，并显示当前待完成心愿数量。随机推荐池合并饭盆菜品和待完成心愿，分类筛选同时作用于两类数据。

推荐卡必须显示来源：

- `饭盆 · 做过 N 次`
- `心愿单 · 还没做过`

同一道菜同时存在于两个库时，优先按 `recipe_id` 去重，其次按规范化菜名和分类去重。加入早餐、午餐或晚餐只生成计划，不完成心愿。

### 3.2 心愿单页

心愿单页包含标题、待完成及已完成数量、菜名搜索框、待完成列表和“已完成心愿”入口。

搜索结果按以下优先级排序：

1. 规范化菜名精确匹配。
2. 别名精确匹配。
3. 菜名或别名包含关键词。

搜索卡片标记“已在心愿单”或“已在饭盆”。同一菜谱不能重复加入待完成心愿。无结果时明确提示更换关键词，不使用 AI 内容冒充正式菜谱。

### 3.3 菜谱详情页

菜谱详情展示分类、人数、预计时间、采购清单、食材用量、制作步骤、来源和许可证。没有可用图片时使用分类图标，不抓取无授权网络图片。

### 3.4 已完成心愿页

已完成心愿按完成时间倒序展示菜名、完成日期、成品照片快照、对应饭盆菜品是否仍存在，以及菜品或菜谱详情入口。

### 3.5 成长日记

日记顶部增加心愿统计卡，显示待完成和已完成数量，点击进入同一个心愿单页面。完成事件进入时间线，文案为“完成心愿：菜名 · 心愿成就 +1”。

## 4. 数据设计

### 4.1 公共菜谱

- `id`
- `name`
- `name_key`
- `category_key`
- `description`
- `servings`
- `estimated_time_minutes`
- `source_name`
- `source_url`
- `source_license`
- `source_path`
- `source_revision`
- `content_hash`
- `image_url`
- `created_at`
- `updated_at`

`recipe_ingredients`：

- `id`
- `recipe_id`
- `sort_order`
- `ingredient_name`
- `amount_value`
- `amount_unit`
- `amount_text`
- `optional`
- `note`

`recipe_steps`：

- `id`
- `recipe_id`
- `sort_order`
- `text`
- `section_name`

`recipe_aliases`：

- `id`
- `recipe_id`
- `alias`

HowToCook 的原始分类通过固定映射归入猪猪食堂六类。无法可靠解析的人数、时间或结构化用量允许为空，但原始可读文本必须保留。

### 4.2 心愿与完成记录

`wishlist_items`：

- `id`
- `owner_id`，本轮兼容空间可为空
- `recipe_id`，后续自建心愿可为空
- `custom_name`
- `name_key`
- `category_key`
- `status`：`pending` 或 `completed`
- `added_at`
- `completed_at`
- `completed_dish_id`
- `created_at`
- `updated_at`

`wishlist_completions`：

- `id`
- `owner_id`
- `wishlist_item_id`
- `recipe_id`
- `completed_dish_id`，删除饭盆菜品时置空
- `added_at_snapshot`
- `completed_at`
- `name_snapshot`
- `image_url_snapshot`
- `created_at`

数据库使用唯一约束阻止同一所有者重复添加同一个待完成菜谱。完成快照与饭盆生命周期分离，删除菜品不会删除或回退完成记录。

### 4.3 饭盆与今日菜单

`dishes` 增加：

- `recipe_id`，可为空
- `wishlist_item_id`，可为空
- `owner_id`，为账号系统预留

`meal_plans` 增加：

- `recipe_id`，可为空
- `wishlist_item_id`，可为空
- `source_type`：`dish` 或 `wishlist`
- `owner_id`，为账号系统预留

饭盆来源的计划保存 `dish_id`；心愿来源的计划保存 `wishlist_item_id` 和 `recipe_id`。一条计划只能拥有一种有效来源。

## 5. 心愿完成事务

用户点击保存饭盆菜品前，服务端按 `recipe_id` 优先、规范化菜名和分类其次，检查待完成心愿。

命中后展示确认框：

> 这道菜已经做过，是否从心愿单移除并记为已完成？

操作：

- 主操作：“移除并记为已完成”。
- 次操作：“暂时保留”。

选择完成时，单个数据库事务依次创建饭盆菜品、更新心愿状态并插入完成快照。任一步失败时全部回滚。选择暂时保留时只创建饭盆菜品，心愿继续保留。

事务成功后播放轻量“心愿已完成 +1”动效，提供“返回饭盆”和“查看已完成心愿”。

## 6. AI 识图设计

### 6.1 稳定返回结构

识图服务统一返回：

```json
{
  "candidates": [
    { "name": "候选菜名", "category": "肉类" }
  ],
  "visibleIngredients": ["可见食材"],
  "provider": "gemini",
  "requestId": "日志追踪标识"
}
```

`candidates` 最多三项。服务端验证候选名称、分类枚举和数组长度，丢弃不合法字段。模型不返回用量和制作步骤。

### 6.2 用户交互

1. 用户拍照或选择成品照片。
2. 客户端压缩后先上传并获得稳定图片地址。
3. 用户主动点击“AI 智能识别菜名”。
4. 页面展示最多三个候选和“都不对，手动输入”。
5. 只有用户点击候选后才填写菜名；已手动修改的菜名或分类不自动覆盖。
6. 可见食材只作为提示，不写入正式用量字段。
7. 候选命中公共菜谱时提供菜谱详情入口，并在保存时关联 `recipe_id`。

图片上传和识图调用分离。AI 超时、失败或返回异常时，图片和表单状态保持不变，用户可以手动完成保存。

### 6.3 公网 Gemini

公网环境使用 Gemini，设置明确超时、最多一次自动重试、返回结构校验和安全错误日志。日志记录请求标识、供应商、耗时和错误类型，不记录图片二进制或完整 Base64。

### 6.4 局域网 Qwen POC

M3 Pro 18GB 电脑使用 Ollama 运行 `qwen3-vl:4b-instruct-q4_K_M`。局域网手机访问电脑上的 Next.js 服务，由 Next.js 服务端调用 `http://127.0.0.1:11434`；手机浏览器不直接访问 Ollama。

模型适配层按环境配置选择供应商：局域网优先 Qwen，Qwen 不可用时回退 Gemini；公网只使用 Gemini。两个供应商必须返回同一结构。

POC 保存每张测试照片的候选、响应时间、失败类型和模型版本，用于后续 50–100 张家庭真实照片评测。本轮不因 POC 结果自动切换公网模型。

## 7. 接口设计

- `GET /api/recipes/search?q=菜名`
- `GET /api/recipes/:id`
- `GET /api/wishlist?status=pending`
- `POST /api/wishlist`
- `DELETE /api/wishlist/:id`
- `GET /api/wishlist/completed`
- `POST /api/wishlist/:id/complete`
- `GET /api/recommendations?category=all`
- `POST /api/uploads/dish-photo`
- `POST /api/recognize`
- `GET /api/recognize/health`，仅在非生产环境返回供应商可用状态

现有 `POST /api/dishes` 扩展为接受可选的 `recipeId`、`wishlistItemId` 和 `completeWishlist`。服务端重新验证心愿归属、状态和菜谱匹配，不能信任客户端直接传入的完成结果。

## 8. 异常处理

- 菜谱导入先解析到暂存结果；存在致命结构错误时不写生产库。
- 重复执行导入按来源版本和内容哈希幂等更新，不产生重复记录。
- 心愿重复添加由接口检查和数据库唯一约束共同拦截。
- 单次心愿完成失败时事务完整回滚。
- 公共菜谱读取失败不阻塞饭盆、今日菜单或手动添加菜品。
- AI 失败不删除图片、不重置表单、不阻止手动保存。
- Qwen 健康检查失败时自动回退 Gemini，并在开发日志标明供应商切换。
- 推荐池任一来源暂时失败时保留另一来源，并向用户展示非阻塞提示。

## 9. 测试设计

### 9.1 自动化测试

- HowToCook Markdown 解析、分类映射、别名和重复导入。
- 搜索的精确菜名、别名、包含匹配和排序。
- 重复心愿的接口检查与数据库约束。
- 心愿待完成、完成、移除和永久快照。
- 删除饭盆菜品后完成记录保留且关联置空。
- 心愿菜加入今日菜单不完成心愿、不增加做过次数。
- 推荐池跨库去重、分类筛选和来源文案。
- AI Top 3 结构校验、超时、一次重试、供应商回退和手动降级。
- 保存饭盆与完成心愿事务任一步失败时全部回滚。
- 现有菜品编辑、删除、图片、菜单和历史统计回归。

API 测试使用隔离的本地 SQLite/libSQL 测试库，不连接生产 Turso，不写入真实用户数据。

### 9.2 手工走查

- 菜单页爱心入口、数量角标、搜索、详情和返回路径。
- 待完成与已完成列表滚动、空状态和删除确认。
- iPhone 与 Android 上的触控目标、底部安全区和页面完整滚动。
- 局域网手机调用 Qwen、本机模型停止后的 Gemini 回退。
- AI 失败后照片预览、手动输入和保存保持正常。
- 完成动效结束后的两个去向均可达。

## 10. 分批交付

### 批次一：公共菜谱

完成固定版本导入工具、数据库表、搜索、结果和详情。导入报告经核对后才写入生产数据库。

### 批次二：心愿单与推荐

完成待完成、已完成、日记统计、今日菜单双来源和跨库推荐去重。

### 批次三：AI 与完成链路

拆分图片上传与识图，改造 Gemini Top 3，并打通保存饭盆、心愿确认、事务和完成动效。

### 批次四：局域网 Qwen POC

安装 Ollama、下载 4B 量化模型、接入供应商适配层并建立家庭照片评测记录。

每个批次单独提交、验证和部署。正式数据库迁移前创建备份；部署后逐项验证首页、菜谱、心愿单、今日菜单、日记、图片和识图接口。

## 11. 验收标准

- 公共菜谱与饭盆分表存储，导入不会产生做过记录。
- 菜名和别名均可搜索，详情展示用量、步骤和来源。
- 同一菜谱不能重复加入待完成心愿。
- 菜单页和日记页都能进入同一个心愿单。
- 已完成心愿永久保留，删除饭盆菜品不会删除完成快照。
- 随机推荐同时读取饭盆与待完成心愿，来源清晰且不重复。
- 心愿菜加入今日菜单不会完成心愿。
- AI 最多返回三个候选，只有用户确认后才填写菜名。
- AI 不生成用量和步骤，失败不影响图片和手动保存。
- 公网 Gemini 和局域网 Qwen 使用相同返回契约。
- 保存饭盆并完成心愿的数据变更具备事务一致性。
- 现有编辑、删除、图片、今日菜单和成长日记回归通过。

## 12. 非本轮范围

- 自定义无菜谱心愿菜的用户入口。
- 将 Qwen 暴露为公网常驻服务。
- 使用 AI 自动生成正式菜谱。
- 家庭成员、邀请和权限。
- 微信登录、短信登录和账号同步实现。
- 根据 POC 自动替换公网 Gemini。
